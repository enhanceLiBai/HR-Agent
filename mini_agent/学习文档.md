## 1. 最小示例：纯文本流式

先看最简情况——**没有工具调用，大模型直接回复文本**。核心就三步：

```python
# ① API 层：加 stream=True
stream = client.chat.completions.create(
    model="deepseek-v4-flash",
    messages=[{"role": "user", "content": "讲个笑话"}],
    stream=True,       # ← 就这一个参数
)

# ② 消费层：逐块读取
for chunk in stream:
    delta = chunk.choices[0].delta
    if delta.content:
        print(delta.content, end="", flush=True)  # 实时打印
```

**效果**：控制台会一个字一个字往外蹦，而不是等 3 秒后整段弹出。

这就是流式输出的全部秘密——**加一个参数，换一种消费方式**。

## 2. 原理拆解

### 2.1 对比：非流式 vs 流式

```
非流式（你现在的写法）
────────────────────────────────────────────
client.chat.completions.create(...)
        │
        │  ⏳ 阻塞等待 3~5 秒...
        │
        └── 返回 response 对象
            response.choices[0].message.content
            → "哈哈，有一天程序员去买菜..."

流式（stream=True）
────────────────────────────────────────────
client.chat.completions.create(..., stream=True)
        │
        └── 立即返回迭代器（不阻塞）
            │
            ├── chunk 1: delta.content = "哈"
            ├── chunk 2: delta.content = "哈"
            ├── chunk 3: delta.content = "，"
            ├── chunk 4: delta.content = "有"
            ├── chunk 5: delta.content = "一"
            ├── ...
            └── chunk N: delta.content = None  （结束标记）
```

### 2.2 关键差异

| | 非流式 | 流式 |
|---|---|---|
| 参数 | 默认 | `stream=True` |
| 返回类型 | `ChatCompletion` 对象 | 迭代器 `<Stream>` |
| 取值位置 | `response.choices[0].message` | `chunk.choices[0].delta` |
| 字段名 | `.message.content`（全量） | `.delta.content`（增量） |
| 内容形态 | 完整字符串 | 逐块碎片，需自己拼接 |
| 阻塞时间 | 等待全部生成完 | 几乎不阻塞 |

### 2.3 Python 生成器的角色

`for chunk in stream` 能逐块读取，是因为 `stream` 对象内部是一个**生成器**。它的大致原理：

```python
# DeepSeek SDK 内部大概是这样的（简化）：
def _stream_response(http_response):
    for line in http_response.iter_lines():
        if line.startswith("data: "):
            data = json.loads(line[6:])  # 解析 SSE 格式
            yield data                     # ← yield，不是 return
```

每次 `yield` 一个 chunk，你的 `for` 循环就拿到来处理。生成完后函数才退出。这就是为什么你能**边收边处理**——`yield` 让函数像水龙头一样，而不是像水桶一样。

## 3. 完整可运行代码

下面是一个**独立可运行**的最小流式应用。新建一个文件 `stream_demo.py` 跑一下就能看到效果。

```python
"""
stream_demo.py —— 流式输出最小演示
====================================
依赖：pip install fastapi uvicorn openai python-dotenv
运行：python stream_demo.py
打开：http://127.0.0.1:8002/chat?msg=讲个笑话
"""
import os, json
from dotenv import load_dotenv
load_dotenv(".env")  # 假设 .env 在当前目录

from openai import OpenAI
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import uvicorn

app = FastAPI(title="Stream Demo")


def stream_chat(user_msg: str):
    """
    生成器函数 —— 流式调用 DeepSeek，逐个产出 chunk

    这是整个流式输出的核心：
    - stream=True 让 API 返回迭代器
    - yield 让 FastAPI 能把每块数据实时推送给浏览器
    """
    client = OpenAI(
        api_key=os.getenv("DEEPSEEK_API_KEY"),
        base_url=os.getenv("DEEPSEEK_BASE_URL", "https://api.deepseek.com/v1"),
    )

    # ── ① 流式调用 ──
    stream = client.chat.completions.create(
        model=os.getenv("DEEPSEEK_MODEL", "deepseek-chat"),
        messages=[{"role": "user", "content": user_msg}],
        stream=True,
    )

    # ── ② 逐块产出 ──
    for chunk in stream:
        delta = chunk.choices[0].delta
        if delta.content:
            # SSE 格式：data: <内容>\n\n
            yield f"data: {delta.content}\n\n"

    # ── ③ 结束标记 ──
    yield "data: [DONE]\n\n"


@app.get("/chat")
async def chat(msg: str = "你好"):
    """
    浏览器访问：http://127.0.0.1:8002/chat?msg=你好

    StreamingResponse 接收一个生成器，
    生成器每 yield 一次，数据就立刻发送给客户端，不等生成器结束。
    """
    return StreamingResponse(
        stream_chat(msg),
        media_type="text/event-stream",   # ← 告诉浏览器这是 SSE 流
    )


if __name__ == "__main__":
    print("🌊 流式 Demo 启动：http://127.0.0.1:8002/chat?msg=讲个笑话")
    uvicorn.run(app, host="127.0.0.1", port=8002)
```

### 怎么验证效果？

浏览器打开 `http://127.0.0.1:8002/chat?msg=讲个笑话`，你会看到文字**一个字一个字蹦出来**。

如果你用 `curl`：
```bash
curl -N http://127.0.0.1:8002/chat?msg=你好
```
因为终端不识别 SSE，你会看到原始格式：
```
data: 你
data: 好
data: ！
data: 我
data: 是
...
data: [DONE]
```

## 4. 进阶：带工具调用的流式

带工具调用时，情况变复杂——大模型可能先吐几个文本 chunk，然后切换到吐 `tool_calls` 的 chunk。

### 4.1 流式 chunk 的完整结构

每个 `chunk.choices[0].delta` 可能包含：

```python
delta.content          # None 或 "今"
delta.tool_calls       # None 或 [{"index": 0, "function": {"name": "calc", "arguments": "{\"expr"}}]
```

大模型在一个回复中可能**交替**产出文本和工具调用：
```
chunk 1: delta.content = "让我"       ← 文本，tool_calls = None
chunk 2: delta.content = "算一下"     ← 文本，tool_calls = None
chunk 3: delta.content = None        ← 开始工具调用！
chunk 3: delta.tool_calls[0].function.name = "calculate"
chunk 3: delta.tool_calls[0].function.arguments = '{"expr'
chunk 4: delta.content = None
chunk 4: delta.tool_calls[0].function.arguments = 'ession": "3+5"}'
chunk 5: delta.content = None        ← 工具调用结束
chunk 5: delta.tool_calls = None
```

### 4.2 处理 tool_calls 需要缓冲区

因为 name 和 arguments 都是分 chunk 来的，你必须**累积拼接**：

```python
def stream_chat_with_tools(user_msg: str):
    client = OpenAI(...)

    stream = client.chat.completions.create(
        model="deepseek-chat",
        messages=[{"role": "user", "content": user_msg}],
        tools=[...],     # 跟非流式一样传
        stream=True,
    )

    # ── 缓冲区 ──
    content_buffer = ""          # 拼文本
    tool_call_buffer = {}        # 拼工具调用，格式：{index: {"name": "...", "arguments": "..."}}

    for chunk in stream:
        delta = chunk.choices[0].delta

        # 处理文本增量
        if delta.content:
            # 先不 yield 给前端，等确定没有 tool_calls 再发
            content_buffer += delta.content

        # 处理工具调用增量
        if delta.tool_calls:
            for tc in delta.tool_calls:
                idx = tc.index                    # tool_calls 可能有多个，用 index 区分
                if idx not in tool_call_buffer:
                    tool_call_buffer[idx] = {"id": tc.id, "name": "", "arguments": ""}

                if tc.function and tc.function.name:
                    tool_call_buffer[idx]["name"] += tc.function.name
                if tc.function and tc.function.arguments:
                    tool_call_buffer[idx]["arguments"] += tc.function.arguments

    # ── 循环结束后判断 ──
    if tool_call_buffer:
        # 有工具调用：把拼好的 tool_calls 返回给调用方
        tool_calls = list(tool_call_buffer.values())
        yield f"data: [TOOL_CALLS] {json.dumps(tool_calls, ensure_ascii=False)}\n\n"
    else:
        # 纯文本：把 buffer 里的内容流式输出
        yield f"data: {content_buffer}\n\n"

    yield "data: [DONE]\n\n"
```

### 4.3 核心思路总结

```
              ┌──────────────────┐
              │   for chunk in   │
              │     stream:      │
              │                  │
              │  delta.content?  │──有──→ 拼到 content_buffer
              │                  │
              │  delta.tool_     │──有──→ 拼到 tool_call_buffer
              │    calls?        │
              └──────────────────┘
                      │
                      ▼
              ┌──────────────────┐
              │  循环结束后判断：  │
              │                  │
              │  tool_call_buffer│──非空──→ 执行工具，再调 API
              │    非空？         │
              │                  │
              │    为空           │──→ 流式输出 content_buffer
              └──────────────────┘
```

## 5. 接入你现有的 Mini Agent

你现在的 `agent.py` 里的 `chat()` 函数是**普通函数**（`return` 最终结果）。要改成流式，变化点：

| 位置 | 原来 | 改成 |
|---|---|---|
| `chat()` 函数类型 | 普通函数 `-> str` | 生成器函数（用 `yield`） |
| API 调用 | `stream=False`（默认） | `stream=True` |
| 返回值取值 | `msg.content`（全量） | 逐 chunk 的 `delta.content`（增量） |
| 最终回复 | `return reply` | `yield reply`（多次 yield） |
| tool_calls 解析 | 一次 `json.loads()` | 先拼 buffer 再 `json.loads()` |
| server.py 路由 | 返回 `dict` / `ChatResponse` | 返回 `StreamingResponse` |

实战建议：
1. 先在 `agent.py` 写一个新的生成器函数 `chat_stream()`，保留原 `chat()`
2. 在 `server.py` 加一个新的路由 `/chat/stream`，用 `StreamingResponse`
3. 两头都跑通后，再决定是否替换掉旧的

## 附录：为什么 SSE 而不是 WebSocket？

| | SSE | WebSocket |
|---|---|---|
| 方向 | 单向：服务端 → 客户端 | 双向 |
| 协议 | 纯 HTTP | 升级为 WS 协议 |
| 实现 | `StreamingResponse` 一行 | 需要 `websockets` 库 |
| 自动重连 | 浏览器原生支持 | 需要自己写 |
| 适用场景 | 服务端推送流式数据 | 实时双向通信（聊天室）|

AI 流式输出只需要**服务端 → 客户端**，SSE 是最轻量的选择。
